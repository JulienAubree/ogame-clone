# Colonization Events Redesign — "Colonie vivante"

**Date:** 2026-04-16
**Status:** Approved

## Summary

Replace the current binary colonization event system (raid/shortage with fixed penalties) with two continuous pressure mechanics:

1. **Resource consumption** — the colony consumes resources continuously; running out halves progression speed
2. **Pirate raids** — real inbound fleets resolved via `simulateCombat()` against a player-stationed garrison

The one-shot missions (consolidate, supply, reinforce) are replaced by free transport and multi-send reinforcements.

---

## 1. Resource Consumption

### Mechanic

The colony consumes resources every worker tick (5 min) to sustain construction.

**Hourly consumption:**
```
minerai_per_hour  = base_minerai  * (1 + scaling_factor * ipc_level)
silicium_per_hour = base_silicium * (1 + scaling_factor * ipc_level)
```

| Config key | Default | Description |
|------------|---------|-------------|
| `colonization_consumption_minerai` | 200 | Base minerai consumption/hour |
| `colonization_consumption_silicium` | 100 | Base silicium consumption/hour |
| `colonization_cost_scaling_factor` | 0.5 | IPC scaling (existing key, reused) |

No hydrogene consumed (reserved for transport/fuel).

### Impact on progression

- Stock sufficient on both resources: progression at normal rate (`passive_rate * difficultyFactor`)
- Stock at 0 on at least one resource: progression **divided by 2**
- The worker deducts resources first, then advances the progress bar in the same tick

### Outpost establishment (replaces consolidate)

- New field: `outpostEstablished` (boolean, default false) on `colonizationProcesses`
- The progress bar does **not advance** until `outpostEstablished = true`
- Triggered when the first resource delivery arrives with at least:
  - `500 * (1 + 0.5 * ipcLevel)` minerai
  - `250 * (1 + 0.5 * ipcLevel)` silicium
- The delivery is a standard `transport` mission — no special mission type
- The `transport` handler must check on arrival: if target planet is `colonizing` and `outpostEstablished = false`, verify the delivered resources meet the threshold and flip the flag

### Supply mechanism

- The `colonize_supply` mission type is **removed**
- Players send standard `transport` missions to the colonizing planet (unlimited sends)
- Resources are added to the planet's stock as with any transport
- No one-shot boost, no tranche system

### Player signaling

**ColonizationProgress page — "Logistique" section:**
- Current stock (minerai / silicium) with visual bars
- Consumption rate displayed (e.g., "-200 minerai/h, -100 silicium/h")
- Time until stockout (e.g., "Stock epuise dans 3h20")
- Status indicator: green (stock OK), orange (< 2h of stock), red (stock at 0, progression slowed)
- When `outpostEstablished = false`: central message explaining the need to send a first convoy, with direct button to fleet send page

**Push notifications:**
- Stock drops below 2h of reserves: "Stocks critiques sur [planete] — envoyez un convoi"
- Stock reaches 0: "Progression ralentie sur [planete] — la colonie manque de ressources"
- On colonization start (colony ship arrival): "Colonisation demarree — envoyez des ressources pour etablir l'avant-poste et maintenir la progression"

## 2. Pirate Raids

### Generation

Raids are generated by the colonization worker at regular intervals.

| Config key | Default | Description |
|------------|---------|-------------|
| `colonization_raid_interval_min` | 3600 | Min seconds between raids (1h) |
| `colonization_raid_interval_max` | 5400 | Max seconds between raids (1h30) |
| `colonization_raid_travel_min` | 1800 | Min travel time in seconds (30min) |
| `colonization_raid_travel_max` | 3600 | Max travel time in seconds (1h) |

New field on `colonizationProcesses`: `lastRaidAt` (timestamp) — tracks the interval between raids.

A raid only generates if `outpostEstablished = true` (no raids before the colony starts).

### Raid as fleetEvent

Each raid creates a `fleetEvent` entry:
- `mission: 'colonization_raid'` (new mission type)
- `phase: 'outbound'`
- `status: 'active'`
- `targetGalaxy/System/Position`: the colonizing planet's coordinates
- `ships`: generated pirate fleet composition
- `departureTime`: now
- `arrivalTime`: now + random travel time

The raid is visible as a hostile inbound fleet on the ColonizationProgress page (countdown timer, fleet composition).

### Pirate fleet scaling

**Target FP formula:**
```
targetFP = base_fp * (1 + scaling_factor * ipc_level) * (1 + stationed_fp_ratio)
```

| Config key | Default | Description |
|------------|---------|-------------|
| `colonization_raid_base_fp` | 50 | Base pirate FP |
| `colonization_raid_stationed_fp_ratio` | 0.3 | Fraction of stationed FP added to pirate base |

- `ipc_level`: player's Imperial Power Center level
- `stationed_fp`: total FP of ships stationed on the colonizing planet
- Pirates have flat combat multipliers (1.0 / 1.0 / 1.0) — no research bonuses
- Fleet composition built from existing pirate templates via `scaleFleetToFP()`

### Combat resolution on arrival

**Case 1 — Garrison present (ships stationed on planet):**
- `simulateCombat()` runs: pirate fleet vs. stationed ships
- **Defense wins:** no damage to progression, no pillage. Debris generated (recyclable). Combat report created.
- **Defense loses:** ship losses applied to `planetShips`, partial resource pillage (33% of stock), reduced progression penalty:
  ```
  penalty = raid_base_penalty * (1 - defense_fp_ratio)
  defense_fp_ratio = pirate_fp_destroyed / total_pirate_fp
  ```
  The more damage the garrison dealt, the lower the penalty.

**Case 2 — No garrison:**
- No combat simulation
- Pillage 50% of stored resources
- Full progression penalty: `raid_base_penalty` (8%)

| Config key | Default | Description |
|------------|---------|-------------|
| `colonization_raid_base_penalty` | 0.08 | Max progression penalty per raid |
| `colonization_raid_no_garrison_pillage` | 0.50 | Pillage ratio when undefended |
| `colonization_raid_garrison_pillage` | 0.33 | Pillage ratio on defense loss (same as PvP) |

### Notifications

- Raid generated: "Flotte pirate detectee en approche de [planete] — arrivee dans Xh Xmin"
- Raid outcome: combat report (same format as existing attack reports, type `colonization_raid`)

### ColonizationProgress page — "Menaces" section

- Shows inbound pirate fleet(s) with countdown timer
- Fleet composition displayed
- Button "Envoyer des renforts" linking to fleet send page

## 3. Mission Reinforce Transformed

### New behavior

- `colonize_reinforce` mission sends ships that **stay on the planet** (inserted into `planetShips`)
- No one-shot limit — player can send reinforcements as many times as needed
- Ships stationed defend automatically during raids
- No passive progression bonus (defense IS the benefit)

### Validation

- At least one ship required (no combat-only restriction — player can station anything)
- Target must be a planet with `status = 'colonizing'`

### On arrival

- `scheduleReturn: false` — ships do not return
- Ships inserted into `planetShips` for the colonizing planet
- Mission report created (type `colonize_reinforce`, lists ships stationed)

### Ship losses during raids

- Destroyed ships are removed from `planetShips`
- Player must send new reinforcements to maintain defense

### After finalization

- When planet transitions to `active`, stationed ships become the permanent garrison

## 4. Removed Mechanics

### Deleted mission types
- `colonize_supply` — replaced by standard `transport`

### Deleted features
- `consolidate` action (local button) — replaced by `outpostEstablished` on first delivery
- `colonizationEvents` table — raids are now `fleetEvents`, shortages replaced by resource consumption
- Auto-resolve mechanic (supply resolves shortage, reinforce resolves raid)

## 5. Migration Strategy

For colonizations in progress at deployment time:
- Finalize/expire all pending `colonizationEvents`
- Set `outpostEstablished = true` (they already started)
- Set `lastRaidAt = now` (grace period before first new raid)
- Drop `colonizationEvents` table after migration
- Remove deleted columns from `colonizationProcesses`

| Metric | Value |
|--------|-------|

| Progression (stock OK) | ~10%/h |
| Progression (stock at 0) | ~5%/h |

| Total resources consumed | ~2000 minerai + 1000 silicium |
| Expected raids | 7-8 over full colonization |
| Raid FP (IPC 0, no garrison) | ~50 FP |

## 7. Full Lifecycle

1. **Launch** — colony ship arrives, planet created as `colonizing`, `outpostEstablished = false`, progression = 0
2. **Outpost** — first transport delivers minimum resources, `outpostEstablished = true`, progression starts
3. **Active phase** — continuous loop:
   - Worker consumes resources, advances progress
   - Raids generated every 1-1.5h as inbound fleetEvents
   - Player sends transports (resources) and reinforcements (ships)
4. **Raid arrives** — combat if garrison present, pillage + penalty if not
5. **Finalization** (progress >= 0.995) — player clicks "Prendre possession" or worker auto-finalizes, planet becomes `active`, stationed ships become garrison
6. **Failure** (progress <= 0) — planet deleted, colony ship returns to origin
